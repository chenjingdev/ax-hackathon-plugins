CS 온보딩 차이점 도우미

채널톡channeltalk-cs-onboarding-deltaCodex 플러그인

새 고객사의 CS 문서(반품·환불 정책·기존 FAQ·상품 안내)를 이커머스 CS 표준 매뉴얼과 대조해, 표준과 다른 것·빠진 것·서로 어긋난 안내·신규 주제만 근거 인용과 함께 리포트하는 온보딩 도구. 모든 판정에 파일:라인 "원문" 인용이 붙는다.

무엇을 하나

채널톡은 AI 상담 에이전트 '알프(ALF)'를 판다. 그런데 알프는 흩어진 지식을 스스로 흡수하지 않는다 — 회사의 지식(FAQ·상품·운영 정책)을 사람이 문서로 정리해 '지식(Knowledge)'에 먹여야 답한다(엑셀·PDF·웹 참조로 학습). [알프 v2 소개] 그래서 신규 고객사마다 흩어진 지식을 사람이 손으로 변환하는 온보딩 손작업이 생긴다.

이 일은 채널톡이 신설한 AX 컨설턴트 직무의 핵심업무로 명문화돼 있다 — 채용공고 문구 그대로: "고객사의 FAQ, 상품 정보, 운영 정책 데이터를 분석하여 ALF가 학습할 수 있는 지식(Knowledge)으로 변환". [AX Consultant 채용공고] 실제 온보딩에서 베리시는 FAQ 65개를 손으로 등록하는 데 2시간 반이 걸렸고 [베리시 케이스], 이랜드이츠는 130개 오프라인 매장 안내를 문서로 입력했다 [이랜드이츠 케이스]. 채널톡 전문가 온보딩 서비스도 "대부분 50만원~150만원 구간"에서 진행되며, 세팅 시 "지식은 고객사에서 제공"한다 [전문가 찾기 도움말].

회사마다 반복되는 이 지식변환이 병목이고, 사람이 문서를 눈으로 훑으면 안내가 서로 어긋나거나 빠진 항목을 놓치기 쉽다. 이 도구는 그 반복 손작업을 친다 — 공통 문의는 표준 매뉴얼이 커버하니 재작성하지 않고, 이 회사만 다른 것·빠진 것·서로 어긋난 안내·신규만 근거와 함께 표면화한다.

돌리면 세 가지가 나온다

세 산출물은 각각 컨설턴트가 온보딩에서 하는 일 하나씩에 대응한다(공통 SAME은 표준이 커버하니 재작성하지 않는다):

  • ① 차이점 리포트 — 검수할 때. 도구는 서로 어긋난 안내(CONFLICT)와 표준과 다른 정책(DIFFERENT)을 파일:라인 "원문"과 함께 짚어준다. 예컨대 같은 문서에서 FAQ 대목은 환불을 "3일", 정책 대목은 "7영업일"이라 말할 때, 사람이 놓칠 뻔한 그 어긋남을 어느 파일 몇 번째 줄인지까지 대며 표면화한다. 컨설턴트는 이 항목만 검수·반영하면 된다.
  • ② 고객사에 물어볼 질문 목록 — 킥오프 미팅 때. 표준에는 있는데 이 회사 문서엔 근거가 없는 슬롯(MISSING)을 "고객사에 확인 필요" 질문으로 모은다. 컨설턴트는 이 목록을 그대로 킥오프에서 물으면 된다.
  • ③ 신규 주제 — 지식을 추가할 때. 표준에 없는 이 회사 고유 주제(EXTRA, 예: 정기구독 해지)를 새 슬롯 후보로 뽑는다. 표준에 없던 항목이니 지식에 새로 추가한다.

시연 등장인물 — 합성 고객사 '아크메샵'

번들 시연의 고객사는 아크메샵이다. 실존 회사가 아니라, 차이점·서로 어긋난 안내·빠진 항목을 검증용으로 일부러 심어둔 합성 데이터다. 아크메샵이 넘긴 자료는 한 파일(acme-shop.md)이고, 그 안에 세 갈래 내용이 담겨 있다:

파일섹션내용
acme-shop.md반품·환불 정책배송·주문·반품·환불·고객센터·정기구독 정책
자주 묻는 질문(FAQ)자주 묻는 질문/답변(환불·배송비·회원탈퇴)
상품·배송 안내배송 범위·반품 배송비 안내

채팅 요약·문서에서는 아크메샵으로 부르고, 코드·경로에서는 acme-shop으로 표기한다. 실제 고객사가 문서를 여러 파일(.md·.csv)로 줘도 그대로 처리한다 — 번들 시연만 한 파일로 통합했다.

어떻게 계산되나 (2계층)

이 도구는 두 계층으로 나뉜다 — AI(에이전트)는 판정을 내리지 않는다. AI는 각 표준 슬롯에 대해 "이 회사 문서에 답이 있나 / 그 답이 표준과 같은 뜻인가"만 읽어 원문 인용과 함께 슬롯별로 추출한다. 최종 분류(5종)·인용이 원문에 실재하는지 검증(환각 걸러냄)·required 슬롯 전수 커버리지는 python 엔진이 계산한다.

입력
고객사 문서(정책·FAQ·상품 — 한 파일 또는 여러 파일)
→
① 슬롯별 추출
AI: 답 유무·표준과 같은 뜻 판단 + 근거가 되는 원문 문장 채집
→
② 인용 실재성 검증
코드: 인용이 원문에 실재하는지 대조(없으면 근거불명으로 강등)
→
③ 결정론 분류
코드: SAME/DIFFERENT/MISSING/CONFLICT/EXTRA 라벨·카운트·커버리지
→
결과
차이점 리포트 + 질문 목록 + 신규 주제

표준 매뉴얼 슬롯 20종은 baseline/ecommerce-cs.json 파일에 박혀 있고(AI가 그때그때 정하지 않는다), 각 슬롯 기준값의 출처(전자상거래법·소비자분쟁해결기준 등 공개 조문 / [SYNTH] 업계 관행 자작)는 baseline/sources.md에 정직하게 구분돼 있다.

무슨 결과가 나오나 (기대 결과)

아크메샵 문서를 표준 매뉴얼 20슬롯과 대조하면 다음이 나온다:

라벨건수뜻 — 컨설턴트 액션
SAME10표준과 동일 — 재작성 불필요
DIFFERENT1이 회사만 다름 — 검수 후 반영
MISSING2문서에 없는 항목 — 고객사에 질문
CONFLICT1서로 어긋난 안내 — 먼저 확인
EXTRA1표준에 없는 신규 — 슬롯 후보
대표 예시
· CONFLICT [refund-period] — acme-shop.md:17 "7영업일 이내에 처리" ↔ acme-shop.md:29 "3일 이내 처리". 같은 문서의 정책 대목과 FAQ 대목이 환불 처리기간을 다르게 말한다(컨설턴트가 훑다 놓칠 뻔한 어긋남).
· DIFFERENT [return-shipping-fee] — 표준은 "단순변심 반품은 구매자 부담", 아크메샵은 acme-shop.md:43 "단순변심의 경우에도 반품 배송비를 무료". 이 회사만의 정책이라 지식에 반영해야 한다.

공통 20슬롯 중 10개가 SAME이라 재작성 0 — 컨설턴트는 어긋남1·차이1·공백2·신규1, 딱 5개만 확인하면 된다. 지어낸 인용은 코드가 원문 대조로 걸러 근거불명 0.

어떻게 쓰나 (3경로)

  • ① Codex 앱 "사용해 보기". 설치 후 앱에서 바로 실행한다. 처음이면 "처음 써봐요, 퀵스타트로 안내해줘"라고 치면 이 문서가 다시 열린다.
  • ② 채팅 복붙. 고객사 문서(정책·FAQ·상품)를 채팅에 첨부하고 "이 고객사 문서들을 이커머스 CS 표준 매뉴얼과 대조해 차이점 리포트를 만들어줘". 시연만 원하면 "합성 고객사 아크메샵 문서로 온보딩 차이점 분석을 시연해줘".

검증 요약

오프라인·순수 python3 stdlib·외부 의존 0. 신뢰의 바닥은 자동 재현이다:

  • 유닛테스트 27개 — 엔진(정규화·분류·그라운딩·리포트)을 회귀 고정. cd src && python -m unittest discover -s engine/tests.
  • 골든 채점(eval/grader.py) — 아크메샵 픽스처로 분류 라벨 일치 · 근거불명 0 · required 슬롯 커버리지 100%를 매번 재확인. python eval/grader.py → PASS.

선 긋기 (비목표)

  • 대고객 상담 챗봇(알프 클론) 아님. 이 도구는 빌드타임(지식 준비·검수)이지 런타임(고객 응대)이 아니다 — 알프의 경쟁자가 아니라 알프에 먹일 지식을 빠르게·검수 가능하게 만드는 보완재.
  • 알프 자동 등록·발송 없음. 모든 출력은 사람 검수 대상(reviewed_by: human-required)이다.
  • 데모는 합성 데이터(아크메샵). 실 상담로그·고객사 사설 데이터에 의존하지 않는다.
채팅에 "퀵스타트"라고 치면 이 문서가 다시 열린다. 산출물은 전부 사람 검수 대상 — 자동 반영·발송 없음.
