Listing Compliance

플러그인 musinsa-listing-compliance · 표시명 “Listing Compliance”

무신사 입점·운영 상품의 표시정보(혼용률·원산지·친환경 표현·시험성적서)가 그린워싱 가이드·표시광고법·성적서와 어긋나는지, 변경 이벤트마다 교차참조·판정·문서화하는 Codex 플러그인. 탐지로 끝내지 않고 정정안과 출력단 3종 초안까지 생성한다(린터 아님).

version 0.2.1 대상 기업 · 무신사 Codex 플러그인 (순수 SKILL + policy-as-code) 런타임 외부 네트워크·API 키 0

무슨 문제를 푸는가

입점 1회 게이트를 “변경 이벤트 상시 집행”으로 반전

누가·언제 쓰나 — 무신사 MD·입점심사팀이 상품의 신규 등록·상세 수정·시즌 갱신·정책(룰) 갱신 이벤트마다 표시정보를 검수한다. 단일 입점 심사 한 번으로 끝나던 검수를, 표시값·정책이 바뀔 때마다 다시 집행하는 상시 컴플라이언스로 바꾼다.

자증: 회사의 입점 검수가 자사 PB의 인조가죽 ‘#에코레더’ 친환경 표현을 수년간 못 걸러 공정위 경고를 받았다(공개 사실 E1).

기둥 A

그린워싱·진화표현 + 소급 재스캔

정적 블랙리스트로 못 잡는 신조어·변형 표현을 ‘금지개념(석유화학 합성소재 ≠ 친환경)의 변형’으로 의미 판단(신뢰도 동반)하고, 룰이 갱신되면 전 코퍼스를 소급 재스캔해 ‘어느 원칙·어느 문장’ 근거와 함께 위반후보를 끌어올린다.

기둥 B

성적서 ↔ 상품 ↔ 클레임 정합성

이종기관 시험성적서를 정규화·퍼지매칭(외래키 product_ref 부재도 브랜드+품번+소재+부위로 연결)해 클레임과 대조하고, 정합성 4분류(정상/부분커버/불일치/외래키불명)로 재판정한다.

두 기둥의 판정은 그대로 반려 사유서·이의제기 재심사 메모·셀러 응대 초안으로 흐른다. 모두 제안이며 사람(MD)이 검토·확정한다.

번들 샘플 카탈로그

설치 즉시 시연 가능한 sample/ — 기대 판정은 sample/EXPECTED.md에 박제

파일시나리오기대 판정
input.product.json 기둥 B 단건 정합성 데모. 자사 PB 무스탕 자켓(690,000원·원산지 중국). 안감을 캐시미어 30 / 폴리 70으로 주장하나 KOTITI 성적서는 아크릴 100 → 안감 불일치. 겉감 ‘에코레더’ 친환경 합성어 + 근거 없는 환경성 표현. BLOCK
rule_set {GW-01, GW-02, GW-03, MX-01, MX-03, TS-02}
BLOCK 4 · WARN 2 · 안감 = 불일치
input.pass.json 오탐 방지 대조군. 면 100% 티셔츠. 몸판 클레임(면 100)이 KOTITI 성적서(면 100)와 일치, 환경성 표현 없음·원산지 표기 있음 → 깨끗한 상품은 막지 않는다. PASS
위반 0건 · 몸판 = 정상
input.rescan.json 정책갱신 소급 재스캔 (킬러 데모). GW-02 확장(에코레더/비건레더/ECO VEGAN LEATHER = 석유화학 인조가죽 → 친환경 표현 금지)을 한 번 갱신하면, 코퍼스 4건을 소급 재평가한다: 자사 PB 인조가죽 자켓 + 타사 인조가죽 토트백 = BLOCK, 면 티셔츠 = 정상 통과, 미등재 친환경 신조어(‘에코퍼’) = 다음 룰 갱신 후보로 상정. BLOCK 2 / PASS 1 / REVIEW 1
코퍼스 4건 · own_brand_flagged = true
EXPECTED.md 골든 기대 출력 박제. 위 세 입력의 기대 판정·플래그된 listing 집합·rule_set·own_brand_flagged·review_candidate를 고정해 둔 정답 서명. 오프라인 grader가 이 서명으로 채점한다. 채점 기준 (판정 아님)

펀치라인: 룰 갱신 한 번 → 무신사가 자사 인조가죽을 소급 적발한다. “무신사조차 못 닿았던 자사 제품을 잡는다.”

사용법 3경로

앱 · 채팅 · CLI

  1. 1

    앱에서 “사용해 보기” 클릭

    ChatGPT/Codex 앱에서 플러그인을 열고 “사용해 보기”를 누르면 첫 프롬프트로 “퀵스타트: 이 플러그인의 용도·구조·샘플 시연 방법을 안내해줘”가 뜬다 — 이 퀵스타트 안내와 함께 번들 샘플 시연을 제안한다.

  2. 2

    채팅에 프롬프트 복붙

    아래를 그대로 붙여넣으면 메인 심사 스킬이 번들 샘플을 검수한다.

    listing-compliance 스킬로 sample/input.product.json을 검수해줘
  3. 3

    CLI (codex)

    마켓플레이스 등록 → 설치 (2줄):

    codex plugin marketplace add https://github.com/chenjingdev/ax-hackathon-plugins
codex plugin add musinsa-listing-compliance@ax-hackathon-plugins

    설치 후 원커맨드 시연 (기대: BLOCK):

    codex exec "listing-compliance 스킬로 sample/input.product.json을 검수해줘"

기대 출력 미리보기

고정 템플릿 리포트 골격 — templates/report.template.md

모든 검수는 아래 고정 골격을 그대로 채워 출력한다(섹션 추가·삭제·재배열 금지). 채울 내용이 없는 슬롯은 (없음)/해당 없음으로 남긴다.

  • §0판정 요약 — 리포트 ID · 검수 모드 · 대상 · 최종 판정 · 적용 규칙 ID · 근거 데이터
  • §1결정론 바닥 — 탐지된 위반(규칙 ID · 심각도 · 부위 · 정합성 · 주장 vs 관측 · 근거 URL)
  • §2에이전트 표면 — 상정(자문 · 하드 BLOCK 아님, 신뢰도·근거 문장 동반)
  • §3소급 재스캔 결과 — 추가된 룰 · 플래그된 listing · 자사 PB 포함 여부 · 정상 통과
  • §4정정안(생성·제안) — 표기·카피·태그·누락 4종
  • §5출력단 3종 초안 — 반려 사유서 · 이의제기 재심사 메모 · 셀러 응대
  • §6결론 + 검증 푸터(기계 판독: report_id · verdict · rule_ids_used · match_status_by_part · own_brand_flagged · reviewed_by)

verdict 체계

BLOCK 게시 보류 권고 (BLOCK≥1) · WARN 주의 (BLOCK 0, WARN≥1) · PASS 위반 0

정합성 4분류 (match_status)

정상(match) · 부분커버(partial) · 불일치(mismatch = 허위기재) · 외래키불명(unlinked · 퍼지 신뢰도 임계 미만 → 에이전트 표면 상정)

제약 · 비목표

이 에이전트가 하지 않는 것

  • 자동 게시·발송 없음. 모든 정정·문서는 제안이며 reviewed_by = "human-MD-required" — 사람(MD)이 검토·확정하기 전 게시·발송하지 않는다.
  • BLOCK = 게시 보류 권고이지 자동 차단이 아니다.
  • OCR·이미지 유사도는 비목표. 시험성적서는 구조화 JSON으로 입력받고, 코어는 인지노동(의미 판단·교차참조·문서작성)에 집중한다.
  • 단일 입점 게이트 아님 · 챗봇/요약봇 아님 · 고객 대면 제품 아님(내부 심사 도구) · 단순 린터 아님 · 법률 자문/최종 판정 권한 없음.
  • 무신사 비공개 코드·외부 임베딩 API 불요 — 런타임 외부 네트워크·API 키 0. 의미 추론은 Codex 자신이 수행한다.
